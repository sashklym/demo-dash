# Widgets Architecture

> Scope: `be/src/modules/widgets`, the widget resource + chart data.

## Purpose

Widgets are the content of a dashboard. Three types: `line`, `bar` (charts), and `text` (editable body). Each belongs to exactly one dashboard.

## Data model

`Widget` (`widgets` table): `id` (uuid), `dashboard_id` (FK, indexed, `ON DELETE CASCADE`), `type`, `row_index` / `col_index` / `size` (its slot on the 3-column grid, indexed with `dashboard_id`), `title`, `text` (null for charts), `seed` (chart data source), `period`, timestamps. `CHECK (col_index + size <= 3)`. The public DTO exposes the slot as `row` / `col` / `size` — `ROW` is a SQLite keyword.

## Chart data — deterministic by seed

Charts don't store their series; they store a `seed`. `GET …/widgets/:id/data?points=N` returns a series generated by a `mulberry32(seed)` PRNG — so the **same** chart renders identically across reloads (the "data restored after refresh" requirement). `POST …/regenerate` assigns a new seed. Text widgets return 400 for `/data`.

## Layout — explicit row/column slots

Widgets occupy `size` contiguous columns starting at `col_index` in `row_index` on a canonical 3-column grid; reading order is `(row_index, col_index)` and there is no separate rank (see [row-based-widget-layout](row-based-widget-layout.md) and `core/place-widget.ts`).

A new widget goes in the **first row from the top** with a contiguous run of `size` free columns, and opens a new row only when no row has one — a free column is not a free *run*. Deleting frees the columns and leaves the hole open for the next create; only a row with nothing left in it collapses, pulling the rows below up with one bulk `UPDATE`. Resizing grows in place when the row still has the room and re-places by first fit otherwise.

`PUT …/widgets/:id/place` drops a widget on a `(row, col)` slot: a free run just moves it (every other hole survives), an occupied slot splices it into reading order and re-packs from that row down. `PUT …/reorder` compacts the whole board into a given order, squeezing every hole out — the small draggable grid's drop handler. The list endpoint pages by **row range** and reports `totalRows`, which is what sizes the virtualized scrollbar.

## Endpoints

`GET/POST /widgets` (list pages by rows: `?fromRow&toRow`, returns `WidgetPage` with `total` + `totalRows`), `PATCH/DELETE /widgets/:id`, `PUT /widgets/reorder`, `PUT /widgets/:id/place`, `GET /widgets/:id/data`, `POST /widgets/:id/regenerate` — all under `/api/dashboards/:key`.
