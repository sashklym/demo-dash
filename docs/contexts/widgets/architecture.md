# Widgets Architecture

> Scope: `be/src/modules/widgets`, the widget resource + chart data.

## Purpose

Widgets are the content of a dashboard. Three types: `line`, `bar` (charts), and `text` (editable body). Each belongs to exactly one dashboard.

## Data model

`Widget` (`widgets` table): `id` (uuid), `dashboard_id` (FK, indexed, `ON DELETE CASCADE`), `type`, `position` (grid order), `title`, `text` (null for charts), `seed` (chart data source), timestamps.

## Chart data — deterministic by seed

Charts don't store their series; they store a `seed`. `GET …/widgets/:id/data?points=N` returns a series generated by a `mulberry32(seed)` PRNG — so the **same** chart renders identically across reloads (the "data restored after refresh" requirement). `POST …/regenerate` assigns a new seed. Text widgets return 400 for `/data`.

## Positioning

New widgets append at `max(position) + 1` within their dashboard. `PUT …/reorder` reassigns contiguous positions from an ordered id list; the list endpoint always returns widgets ordered by `position`.

## Endpoints

`GET/POST /widgets`, `PATCH/DELETE /widgets/:id`, `PUT /widgets/reorder`, `GET /widgets/:id/data`, `POST /widgets/:id/regenerate` — all under `/api/dashboards/:key`.
