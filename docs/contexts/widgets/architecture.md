# Widgets Architecture

> Scope: `be/src/modules/widgets`, the widget resource + chart data.

## Purpose

Widgets are the content of a dashboard. Three types: `line`, `bar` (charts), and `text` (editable body). Each belongs to exactly one dashboard.

## Data model

`Widget` (`widgets` table): `id` (uuid), `dashboard_id` (FK, indexed, `ON DELETE CASCADE`), `type`, `rank` (fractional order key, indexed with `dashboard_id`), `title`, `text` (null for charts), `seed` (chart data source), timestamps.

## Chart data — deterministic by seed

Charts don't store their series; they store a `seed`. `GET …/widgets/:id/data?points=N` returns a series generated by a `mulberry32(seed)` PRNG — so the **same** chart renders identically across reloads (the "data restored after refresh" requirement). `POST …/regenerate` assigns a new seed. Text widgets return 400 for `/data`.

## Positioning — fractional ranking

Widgets sort by `rank ASC`, a fractional string key (see [fractional-widget-ranking](fractional-widget-ranking.md) and `core/fractional-index.ts`). New widgets append after the last rank. `PUT …/widgets/:id/position` moves one widget to a target index by rewriting its rank between the new neighbors — a single-row write, used by the virtualized grid. `PUT …/reorder` re-spreads fresh ranks from a full ordered id list, used by the small draggable grid. The list endpoint is paged (see [paginated-widget-list-api](paginated-widget-list-api.md)) and always ordered by `rank`.

## Endpoints

`GET/POST /widgets` (list is paged: `?offset&limit`, returns `WidgetPage`), `PATCH/DELETE /widgets/:id`, `PUT /widgets/reorder`, `PUT /widgets/:id/position`, `GET /widgets/:id/data`, `POST /widgets/:id/regenerate` — all under `/api/dashboards/:key`.
